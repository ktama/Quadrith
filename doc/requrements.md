# 3軸タスク管理アプリ 仕様書(v1.1)

## 1. コンセプト

重要度・緊急度・状態の3軸でタスクを管理するWindows向けデスクトップアプリ。
メイン画面は重要度×緊急度の2Dマトリクスとし、第3軸の「状態」は色とフィルタで表現する。
アイゼンハワーマトリクスの「どれからやるか」と、カンバンの「どこまで進んだか」を1画面で扱えることが価値。

## 2. 技術スタック

| 項目           | 選定                                | 理由                               |
| -------------- | ----------------------------------- | ---------------------------------- |
| フレームワーク | Tauri 2.x                           | 軽量(~10MB)・低メモリで常駐に向く  |
| フロントエンド | React + TypeScript                  | D&D実装が容易、型安全              |
| D&Dライブラリ  | dnd-kit(または自前のpointer events) | 自由座標配置に対応しやすい         |
| 状態管理       | Zustand                             | 小規模に適したシンプルさ           |
| DB             | SQLite(tauri-plugin-sql)            | ローカル完結、将来の統計集計に有利 |
| スタイリング   | Tailwind CSS                        | 任意。好みで変更可                 |

※ 自由座標へのドラッグ配置は、dnd-kitのソータブル機能より「pointerイベント + 座標の正規化保存」の自前実装の方が素直に書ける場合が多い。

## 3. データモデル

```sql
-- スキーマバージョン管理(マイグレーション用)
PRAGMA user_version = 1;

CREATE TABLE tasks (
  id          TEXT PRIMARY KEY,        -- UUID
  title       TEXT NOT NULL,
  memo        TEXT DEFAULT '',
  importance  REAL,                    -- 0.0〜1.0(連続値、縦軸)。NULL = インボックス(未仕分け)
  urgency     REAL,                    -- 0.0〜1.0(連続値、横軸)。NULL = インボックス(未仕分け)
  status      TEXT NOT NULL
              CHECK (status IN ('todo','doing','pending','waiting','done')),
  due_date    TEXT,                    -- 'YYYY-MM-DD'(日付のみ。時刻は持たない)
  review_at   TEXT,                    -- 'YYYY-MM-DD' 再確認日。保留・待ちタスクの死蔵防止用
  created_at  TEXT NOT NULL,           -- ISO 8601(UTC)
  updated_at  TEXT NOT NULL,           -- ISO 8601(UTC)
  completed_at TEXT,                   -- 完了日時(アーカイブ判定用)
  deleted_at  TEXT,                    -- 論理削除日時。NULL = 有効
  CHECK ((importance IS NULL) = (urgency IS NULL))  -- 座標は両方あるか両方ないか
);

CREATE INDEX idx_tasks_status  ON tasks(status)  WHERE deleted_at IS NULL;
CREATE INDEX idx_tasks_due     ON tasks(due_date) WHERE deleted_at IS NULL;

CREATE TABLE tags (
  id    TEXT PRIMARY KEY,
  name  TEXT NOT NULL UNIQUE,
  color TEXT NOT NULL                  -- HEX
);

CREATE TABLE task_tags (
  task_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  tag_id  TEXT NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
  PRIMARY KEY (task_id, tag_id)
);
```

設計ポイント:
- importance / urgency は **0.0〜1.0の正規化座標**で保存。ウィンドウサイズに依存せず、象限判定は `>= 0.5` で行える
- **座標がNULLのタスクはインボックス(未仕分け)**として扱う。マトリクスへドラッグした時点で座標が付く
- 状態はステータス文字列のenum。5状態: 未着手 / 進行中 / 保留 / 待ち / 完了
- **削除は論理削除**(`deleted_at`)。誤操作からの復元と統計の整合性を担保する。30日経過した論理削除レコードは起動時に物理削除
- `due_date` / `review_at` は**日付のみ(ローカル日付)**。通知は「当日の指定時刻(設定可、初期値9:00)」に発火
- **スキーマ管理**: `PRAGMA user_version` でバージョン管理し、起動時に現バージョンとの差分マイグレーションを順次適用。任意パスのDB(古いバージョンの可能性あり)を開く仕様のため必須

## 4. 画面仕様

### 4.1 メイン画面(マトリクスビュー)

- 縦軸: 重要度(上が高)、横軸: 緊急度(右が高)
- 中央に十字の境界線、4象限に薄いラベル表示
  - 右上: 「今すぐやる」 / 左上: 「計画する」 / 右下: 「さばく・任せる」 / 左下: 「やめる候補」
- タスクカード: タイトル + 状態色 + 期限バッジ(期限接近で警告色)
- カードをドラッグして任意の位置に配置 → importance/urgencyを即時保存
- カードクリックで詳細パネル(右サイド)を開く
- **インボックスレーン**: マトリクス下部に未仕分けタスク(座標NULL)を横並び表示。レーン→マトリクスへのドラッグで仕分け、逆方向のドラッグで座標を外して差し戻しも可能
- **カードの重なり処理**:
  - 保存座標は変えずに、描画時のみ重なりを検出して放射状に最小限ずらす(衝突回避レイアウト)
  - ホバー/選択中のカードを最前面に表示
  - 同一箇所に4枚以上密集した場合は「+N」のクラスタ表示にまとめ、クリックで吹き出し展開

### 4.2 状態の表現(第3軸)

| 状態   | 色(案) | マトリクス上の扱い           |
| ------ | ------ | ---------------------------- |
| 未着手 | グレー | 表示                         |
| 進行中 | 青     | 表示(枠を強調)               |
| 保留   | 黄     | 表示(半透明)                 |
| 待ち   | 紫     | 表示(半透明)                 |
| 完了   | 緑     | 完了後24hで非表示→アーカイブ |

- ヘッダーに状態フィルタ(トグルチップ)。複数選択可
- カード上の右クリックメニューまたは詳細パネルから状態変更
- 補助ビューとして「状態別カンバン」への切替タブを用意(同じデータの別表示)

### 4.3 詳細パネル

- タイトル、メモ(Markdown可だとなお良い)、期限日、タグ、状態
- **再確認日(review_at)**: 状態を「保留」「待ち」に変更した際に設定を促す。当日になったら通知し、死蔵を防ぐ
- 作成日・更新日の表示、削除ボタン
- **削除は論理削除**とし、削除直後に「元に戻す」付きトーストを5秒表示(undo対応)

### 4.4 クイック追加

- グローバルホットキー(例: `Ctrl+Shift+Space`)で入力ポップアップ
- タイトルだけ入力してEnter → **インボックスレーンに未着手で追加**(座標NULL)
- 後からインボックス→マトリクスへドラッグして仕分ける運用(GTDの受信箱方式)

### 4.5 アーカイブビュー

- 完了から24h経過したタスクの一覧(完了日の新しい順、検索・タグ絞り込み可)
- 各行に「復元」ボタン: 状態を進行中に戻し、元の座標でマトリクスへ再表示
- 論理削除したタスクも「ごみ箱」タブで表示し、復元 or 完全削除が可能(30日で自動消去)
- フェーズ3の統計機能はこのデータを集計対象とする

## 6. 非機能要件

- 起動時間: 2秒以内
- 常駐時メモリ: 100MB以下を目標(Tauriなら余裕)
- データはすべてローカル保存。外部通信なし
- DBファイルの保存先はユーザーが任意に設定可能(詳細は §7)。初期値は `%APPDATA%/<app名>/tasks.db`

## 7. 設定機能

### 7.1 設定の保存方式(2層構成)

DBパス自体はDBに保存できない(起動時にDBを開く前にパスが必要)ため、設定を2層に分ける。

| 層                   | 保存先                                  | 内容                                         |
| -------------------- | --------------------------------------- | -------------------------------------------- |
| ブートストラップ設定 | `%APPDATA%/<app名>/settings.json`(固定) | DBファイルパス、ウィンドウ位置・サイズ       |
| アプリ設定           | SQLite内 `settings` テーブル            | DBと一緒に持ち運びたい設定(色、表示設定など) |

```json
// settings.json の例
{
  "dbPath": "D:/Dropbox/tasks/tasks.db",
  "window": { "x": 100, "y": 100, "width": 1280, "height": 800 }
}
```

```sql
CREATE TABLE settings (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL   -- JSON文字列
);
```

### 7.2 設定画面の項目(案)

**データ**
- DBファイルの保存先(現在パス表示 + 「変更」ボタン → フォルダ選択ダイアログ)
- 「DBファイルの場所を開く」ボタン(エクスプローラで表示)
- 自動バックアップの保持世代数(初期値: 3)と保存先(初期値: DBと同じフォルダの `backups/`)

**表示**
- 状態ごとの色のカスタマイズ
- 完了タスクをマトリクスから消すまでの時間(初期値: 24時間)
- テーマ(ライト/ダーク/システム連動)

**動作**
- クイック追加のグローバルホットキー
- Windows起動時に常駐開始するか
- 閉じるボタンの挙動(終了 or トレイへ最小化)
- 通知の発火時刻(期限日・再確認日の通知。初期値: 9:00)

### 7.3 DBパス変更時の挙動

「変更」でフォルダ選択後、状況に応じて以下を分岐:

1. **移動先にDBが存在しない** → 確認ダイアログ「現在のデータを新しい場所へ移動しますか?」
   - はい: 現在のDBをコピー → 新パスで開き直し → 成功確認後に旧ファイル削除
   - いいえ(新規作成): 空のDBを新パスに作成
2. **移動先に既存のDBがある** → 「既存のデータを開く / 上書きする」を選択させる(既定は開く)
3. 切替前に必ず旧DBへの書き込みをフラッシュし、切替失敗時は旧パスに自動ロールバック

### 7.4 設計上の注意

- DBパスをDropbox等のクラウド同期フォルダに置けば、簡易的なバックアップ・複数台共有が可能(同時起動は非対応と明記)
- 起動時にDBファイルが見つからない場合: エラーで落とさず「ファイルを探す / 新規作成 / 既定の場所に戻す」の選択ダイアログを表示
- SQLiteの一時ファイル(-wal, -shm)も同フォルダに作られるため、移動時は3ファイルセットで扱うか、WALチェックポイント後に単一ファイル化してから移動する
- **自動バックアップ**: 起動時に `VACUUM INTO`(または `backup` API)でDBを `backups/tasks_YYYYMMDD_HHMMSS.db` にコピーし、設定世代数を超えた古いものを削除。単純なファイルコピーはWAL書き込み中だと不整合の恐れがあるため避ける
- **マイグレーション直前にもバックアップを必ず取得**し、失敗時はバックアップから復元して旧バージョンのまま起動を継続する

## 8. 検討済み・保留事項

- **3D表示**: 操作性が悪化するため不採用。状態は色+フィルタで表現
- **重要度・緊急度の段階入力**: 連続値ドラッグを採用。数値の直接入力UIは不要
- **サブタスク**: 今回は対象外(必要になったらフェーズ3以降で検討)
- **クラウド同期**: 自分専用・1台運用のため対象外。複数台が必要になったらDBファイルをクラウドストレージ経由で同期する案も
